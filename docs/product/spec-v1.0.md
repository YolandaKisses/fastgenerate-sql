# 企业内部 SQL 问答桌面应用 Spec

**版本**：Spec v1.0  
**状态**：已确认发布版

## 1. Spec 范围定义

本 spec 仅覆盖首版 MVP，对应已确认的 PRD v1.1。  
本 spec 只定义产品行为、页面要求、状态变化、异常处理和验收口径。  
本 spec 不进入实现方案、代码结构、数据库表设计细节和开发排期。

## 2. 系统定位

系统是一款企业内部自用的桌面 SQL 问答工具。  
系统帮助用户在手动选定数据源的前提下，通过自然语言生成 SQL，并在用户确认后执行只读查询。  
系统不是数据库管理工具，不承担写入、建表、改表、运维、发布、审批等职责。  
系统不是企业知识库，也不承担通用文档问答能力。

## 3. 首版子系统拆分

1. 数据源管理
2. Schema 同步与备注管理
3. SQL 问答
4. SQL 执行与结果展示
5. 本地审计日志

## 4. 全局业务原则

- 所有配置均保存在本机，不依赖中心化服务。
- 用户必须先手动选择数据源，系统不得自动切换数据源。
- 系统不得自动执行模型生成的 SQL。
- 所有执行行为都必须经过“只读校验”与“用户确认”两个关口。
- 首版只支持 OpenAI 兼容协议模型接入。
- 首版不使用 embedding。
- Schema 召回仅基于当前数据源内的表名、字段名、原始备注、本地补充备注做关键词与规则检索。
- 首版结果展示以表格为主，不要求图表。
- 本地审计日志仅用于设备内回看与排查，不支持导出。

## 5. 本地存储规则

首版所有配置、Schema 元数据、本地补充备注和审计日志均保存在应用的本机数据目录中。  
系统采用本地嵌入式存储统一管理，使用 `SQLite` 作为唯一主存储。  
系统不依赖外部配置中心，不要求用户手动管理这些文件。  
`Obsidian` 不作为首版运行时主存储。

## 6. 核心业务对象定义

### 6.1 数据源

一条本地保存的数据库连接配置，包含数据库类型、连接参数、显示名称、可用状态等。

### 6.2 Schema

从数据库同步得到的元数据集合，包含表、字段、字段类型、数据库原始备注。

### 6.3 本地补充备注

用户在本机对表或字段补充的业务解释，与数据库原始备注分离存储，不回写数据库。

### 6.4 生成 SQL

模型根据自然语言问题和召回到的 Schema 生成的候选 SQL，在用户确认前不可自动执行。

### 6.5 审计日志

本地留痕记录，包含提问、数据源、生成 SQL、是否执行、执行状态、耗时、错误信息。

## 7. 数据源管理规则

### 7.1 模块职责

- 新增数据源
- 编辑数据源
- 删除数据源
- 测试连接
- 展示连接状态
- 触发 Schema 同步
- 展示最近同步结果和可用状态

### 7.2 状态定义

- `draft`
- `connection_failed`
- `connection_ok`
- `sync_failed`
- `ready`
- `stale`

### 7.3 核心规则

- 未填写必填字段不得保存。
- 保存成功不等于“可问答”。
- 只有完成至少一次成功 Schema 同步后，数据源才可进入 `ready`。
- 编辑已存在的数据源关键连接信息后，应重新标记为 `stale`。
- 删除数据源时，应同时删除与该数据源相关的本地 Schema 索引和补充备注。
- 历史审计日志应保留文本快照，不因数据源删除而丢失可读性。

## 8. Schema 同步与备注管理规则

### 8.1 核心规则

- Schema 同步不调用 LLM。
- 通过数据库系统表或信息视图直接获取元数据。
- 必须区分“数据库原始备注”和“本地补充备注”。
- 本地补充备注不得回写数据库。
- 同步数据库原始备注时，不得覆盖本地补充备注。
- 若同步结果为空，系统不得将该数据源标记为 `ready`。

### 8.2 同步状态定义

- `never_synced`
- `syncing`
- `sync_success`
- `sync_failed`
- `sync_partial_success`

### 8.3 同步失败规则

首版采用保守策略：  
只有最近一次同步成功的数据源才可问答。  
同步失败后默认禁止继续问答。

## 9. SQL 问答规则

### 9.1 问答前置条件

- 已手动选择数据源
- 数据源状态允许问答
- 当前数据源已有可用 Schema 数据
- 模型配置完整可用

### 9.2 输出类型

问答模块首版只允许两种主输出：

- `clarification`
- `sql_candidate`

### 9.3 澄清优先规则

当出现以下情况时，系统必须先澄清而不是生成 SQL：

- 多个表都可能满足问题目标
- 同一业务词可能映射多个字段
- 统计口径不明确
- 时间边界不明确
- 召回结果无法稳定收敛
- 问题语义过于模糊

### 9.4 LLM 约束机制

系统在调用第三方 LLM 生成 SQL 时，必须通过“三层机制”控制模型行为：

1. 程序前置规则判定
2. Prompt 强约束
3. 返回结果结构校验

系统不得将歧义处理完全交由第三方模型自由发挥。  
即使模型直接返回 SQL，只要当前上下文仍处于高歧义状态，系统也必须拦截该结果。

## 10. SQL 执行与结果展示规则

### 10.1 核心规则

- 所有 SQL 在执行前都必须经过只读校验。
- 系统永不自动执行 SQL。
- SQL 只有在用户手动点击执行后才可进入执行阶段。
- 校验失败的 SQL 必须禁止执行。
- 结果展示以表格为主。
- 若结果为空，应明确展示“无结果”。

### 10.2 首版安全边界

首版必须禁止：

- `INSERT`
- `UPDATE`
- `DELETE`
- `DROP`
- `ALTER`
- `TRUNCATE`
- `CREATE`
- 多语句执行
- 有副作用的存储过程或等价调用

### 10.3 结果保护策略

- 最大返回行数：`500`
- 最大执行时长：`15 秒`

首版不限制只读 SQL 的查询结构复杂度。  
可执行边界由“只读性、结果规模、执行时长、多语句限制”共同控制，而不是由 Join 数量或子查询复杂度控制。

### 10.4 手工编辑规则

首版不开放 SQL 手工编辑执行。

## 11. 本地审计日志规则

### 11.1 日志记录原则

- 日志记录是默认能力，不依赖用户手动开启。
- 日志用于回看、排查、复现问题。
- 首版日志仅本地查看，不导出、不远程同步。

### 11.2 建议记录字段

- 时间
- 数据源名称快照
- 用户问题文本
- 是否触发澄清
- 澄清问题内容
- 生成 SQL
- 是否执行
- 执行结果状态
- 耗时
- 错误信息摘要

## 12. 页面定义

### 12.1 问答主页面

目标：完成“选数据源 -> 提问 -> 澄清 -> 生成 SQL -> 内联确认 -> 执行 -> 查看结果”的单页闭环。

关键模块：

- 数据源选择区
- 问题输入区
- 当前状态提示区
- 澄清问题区
- SQL 候选展示区
- SQL 校验结果区
- 内联确认区
- 结果表格区

### 12.2 数据源管理页

目标：管理本地数据源配置，并明确数据源当前状态。

### 12.3 Schema / 备注管理页

目标：查看元数据，并维护本地补充备注。

### 12.4 模型配置页

目标：配置 OpenAI 兼容模型，使问答流程可用。

### 12.5 审计日志页

目标：在本地查看历史问答与执行记录，用于回看和排查。

## 13. 异常处理原则

- 所有失败必须给出用户可理解的错误反馈。
- 连接失败、同步失败、模型失败、SQL 校验失败、SQL 执行失败都要分开定义。
- 失败后尽量保留用户输入和上下文。
- 模型生成成功但 SQL 校验失败时，仍然保留生成 SQL 供用户查看，但禁止执行。
- 日志写入失败不能导致主流程静默成功；至少要提示“操作成功，但日志记录失败”。

## 14. 全局验收标准

- 支持 `MySQL / PostgreSQL / Oracle`
- 支持本机管理数据源
- 支持 Schema 同步
- 支持原始备注与补充备注分离
- 支持当前数据源内问答
- 歧义场景先澄清
- 只支持生成只读 SQL 候选
- 永不自动执行 SQL
- 用户确认后才执行
- 结果以表格展示
- 本地日志可查看
- 首版仅支持 OpenAI 兼容模型接口
- 首版不依赖 embedding
