# 企业内部 SQL 问答 Web 应用 PRD

**版本**：PRD v1.2  
**状态**：当前工程基线版  
**更新时间**：2026-05-04

## 1. 背景

企业内部已经沉淀了大量业务数据库，但日常取数仍然依赖会写 SQL 的人员，导致查询效率低、沟通成本高、响应慢。

本项目提供一款企业内部自用的 Web SQL 问答工具。用户选择数据源后，用自然语言提问；当存在符合问答条件的数据源时，工作台可默认选中第一个可问答数据源。系统以 SQLite 中的结构化 Schema 元数据作为问答主检索层，按需补充 Obsidian 本地知识库卡片中的业务说明，再由 Hermes 在受控上下文内完成歧义判断、关系补全和只读 SQL 候选生成。系统展示 SQL 并完成安全校验，只有用户手动确认后才执行。

产品聚焦“数据源内 SQL 问答闭环”，不做通用聊天、不做 BI 平台、不做自动执行。

## 2. 目标

- 用户可在 Web 应用中选择一个数据源，并输入自然语言问题；工作台可默认选中第一个已通过连接测试且知识库同步成功的数据源。
- 维护人员可配置 `MySQL / PostgreSQL / Oracle` 数据源，并同步 Schema 元数据。
- 系统支持区分“数据库原始备注”和“本地补充备注”。
- 系统可将当前数据源的 Schema 与备注同步为本地 Obsidian 知识库卡片。
- 系统先从 SQLite 检索相关表、字段和结构化约束，必要时读取相关 Obsidian 表卡片，再通过本地 Hermes CLI 返回澄清问题或 SQL 候选。
- 系统展示生成 SQL 与校验状态，由用户手动确认后再执行。
- 系统执行 SQL 后返回表格结果。
- 系统在本地保留问答、SQL、执行状态、耗时、错误信息和知识库使用信息，便于回看和排查。
- 系统提供本地运行设置，用于配置 Hermes CLI 路径和 Obsidian 知识库根目录。
- 系统提供本地登录入口，业务页面和业务接口必须在登录后访问。

## 3. 非目标

- 不做 workspace、多租户、RBAC 权限体系。
- 不做用户管理后台、角色权限矩阵和组织级账号体系。
- 不做刷新 Token、图形验证码和强制 Cookie 登录态。
- 不做自动推荐数据源。
- 不做跨数据源联合查询。
- 不做图表为主的分析体验。
- 不做数据库写入、DDL、DML 操作。
- 不做外部 OpenAI 兼容模型配置。
- 不做 embedding 检索。
- 不做自动执行 SQL。
- 不做集中化审计平台和日志导出。
- 不做单表单独重新生成、Markdown 增量 diff、应用内 Markdown 编辑、图片/图表/附件同步和多知识库切换管理。

## 4. 目标用户

- 内部分析、运营、产品、财务等有取数需求、但不熟悉 SQL 的员工。
- 少量维护人员负责配置数据源、同步表结构、维护补充备注和同步本地知识库。

## 5. 用户场景

1. 用户通过本地账号登录系统。
2. 维护人员配置“销售库”，测试连接成功。
3. 系统同步表、字段、字段类型、数据库原始备注。
4. 维护人员补充业务口径备注，并同步到 Obsidian 知识库。
5. 知识库同步成功后，用户打开工作台，系统默认选中第一个可问答数据源，或用户手动选择“销售库”。
6. 用户输入“上周华东区销售额是多少”。
7. 系统从 SQLite 检索相关 Schema Context，并按需读取相关 Obsidian 表卡片。
8. Hermes 在受控上下文内判断口径是否清楚；若不清楚，系统先返回澄清问题，用户补充后继续生成 SQL。
9. 系统返回 SQL 候选并完成只读校验。
10. 用户点击“确认并执行”。
11. 系统返回表格结果，并写入本地审计日志。

## 6. 核心流程

1. 用户进入登录页，使用本地账号登录。
2. 配置本地运行设置：Hermes CLI 路径、Obsidian 知识库根目录。
3. 维护人员新增或编辑数据源。
4. 系统测试数据库连接，并展示连接状态。
5. 系统同步 Schema 元数据到本地 SQLite。
6. 维护人员维护表级和字段级补充备注。
7. 系统按数据源生成 Obsidian 知识库目录和表知识卡片。
8. 用户在工作台确认默认数据源，或手动选择其他可问答数据源。
9. 用户输入自然语言问题。
10. 系统校验当前数据源是否有可用知识库，并提示过期风险。
11. 系统基于当前问题和最近对话从 SQLite 检索相关 Schema Context，补充相关 Obsidian 表卡片。
12. Hermes CLI 在受控上下文内生成结构化结果。
13. 系统返回 `clarification` 或 `sql_candidate`。
14. 对 SQL 候选进行只读校验和 Schema 字段校验，展示校验结果。
15. 用户确认后执行 SQL。
16. 系统限制返回行数和执行时间，并展示表格结果。
17. 系统用 `audit_log_id` 绑定生成与执行，更新本地审计日志。

## 8. 边界条件

- 所有配置采用单机本地保存。
- SQLite 是本地配置、Schema、备注、任务状态和审计日志的主存储。
- 普通用户名密码连接模式下，数据源数据库密码使用本地 AES-GCM 密钥加密后保存到 SQLite；服务启动时会迁移旧的明文数据源密码。
- 系统首次启动自动初始化本地管理员账号：用户名称 `系统管理员`，账号 `admin`，密码 `888888`。
- 登录密码传输使用 RSA 加密，服务端只保存密码哈希。
- 登录成功后端返回 Token，前端保存 Token 并在普通请求和流式请求中携带 `Authorization: Bearer <token>`。
- 未携带 Token 或 Token 无效时，前端回到登录页，后端业务接口返回 `401`。
- 登录成功默认进入数据源配置页。
- 登录成功和失败都写入本地登录日志，便于设备内排查。
- Obsidian 是人类可读的业务说明与表卡片载体，不替代 SQLite 主存储，也不是问答主检索层。
- 工作台可默认选中第一个已通过连接测试且知识库同步成功的数据源；系统不得根据用户问题内容自动切换数据源。
- 工作台只展示并允许选择已通过连接测试且知识库同步成功的数据源。
- 系统永不自动执行 Hermes 生成的 SQL。
- 所有 SQL 默认限制为只读查询。
- 首版结果展示以表格为主。
- 本地审计日志仅在当前设备内查看。
- Hermes CLI 和 Obsidian 路径可在本地设置页配置。

## 9. Schema 与知识库说明

Schema 同步不调用 Hermes。系统通过数据库系统表或信息视图读取：

- 表名
- 字段名
- 字段类型
- 数据库原始表备注
- 数据库原始字段备注

用户可维护本地补充备注。补充备注与数据库原始备注分离存储，不回写数据库。

知识库同步会基于 SQLite 中的 Schema 与备注，为每个表生成 Obsidian Markdown 知识卡片。Hermes 可为表用途、核心字段、关联关系和注意事项生成辅助说明；字段明细始终来自结构化 Schema，不采用模型自由生成结果。

知识库按数据源输出到本地运行设置中的 `obsidian_vault_root`，包含数据源总览页和单表详情页。重复同步同一数据源时采用确定性覆盖策略，不保留历史快照目录。

已完成知识库同步的数据源如果再次同步 Schema，系统必须将知识库同步状态改回非成功态，提示用户重新同步知识库。重新同步 Schema 时，数据库中已删除的表和字段应从 SQLite 中清理；对应已删除表的 Obsidian Markdown 文件也应删除，避免旧知识继续被检索。

## 10. 问答链路说明

系统问答链路以 SQLite 中的结构化 Schema 为事实来源和主检索层。后端先根据当前问题和最近对话检索相关表、字段、类型、备注和约束，形成 Schema Context；再按相关表读取对应 Obsidian Markdown 片段，作为业务语义补充。Hermes 只在这些受控上下文内做歧义判断、关系补全和 SQL 候选生成。

Hermes 输出必须是 JSON，且只允许：

- `clarification`
- `sql_candidate`

当问题存在歧义、统计口径不明确、表或字段无法稳定判断时，Hermes 应优先返回澄清问题。后端在多轮澄清时保留最近对话并继续使用当前问题加历史上下文检索 Schema，避免用户只回复 `A`、`是` 等短文本时丢失上一轮表结构上下文。

## 11. 异常与状态

- Hermes CLI 不存在、不可执行、超时或返回非 JSON 时，系统必须给出明确错误。
- Obsidian 目录不存在或不可写时，知识库同步必须失败并提示原因。
- 单表知识卡片生成失败时，不中断整个同步任务。
- 知识库同步失败时保留已经生成的部分文件，不主动回滚；再次同步时覆盖同一数据源目录。
- 知识库同步任务支持 `pending / running / completed / partial_success / failed`。
- Schema 更新晚于知识库任务完成时间时，工作台提示知识库可能已过期。
- SQL 校验失败时禁止执行。只读校验或 Schema 字段校验出现阻断错误时，系统返回明确错误；无法确认的未限定字段问题降级为 warning。
- 日志写入失败时，主流程应提示“操作成功，但日志记录失败”。

## 12. 验收标准

- 可新增、编辑、删除并测试 `MySQL / PostgreSQL / Oracle` 数据源。
- 首次启动可自动创建 `admin / 888888` 管理员账号。
- 登录成功后返回 Token，并默认进入数据源配置页。
- 无 Token 或无效 Token 不可访问业务页面和业务接口。
- 前端普通请求与流式请求均携带 Token，流式能力不得退回前端轮询。
- 编辑关键连接信息后，数据源可标记为 `stale`。
- 系统可同步表结构、字段类型和数据库原始备注。
- 系统支持维护本地补充备注，并与原始备注区分展示。
- 系统可配置 Hermes CLI 路径和 Obsidian 知识库根目录。
- 系统可检测 Hermes 是否可用、Obsidian 目录是否存在且可写。
- 系统可完成知识库同步，并展示成功、部分成功和失败状态。
- 用户可使用默认选中的首个可问答数据源，或手动选择已通过连接测试且知识库同步成功的数据源，并完成自然语言到 SQL 的查询闭环。
- 歧义场景先澄清，澄清后可继续生成 SQL。
- 系统能展示生成 SQL 和 SQL 校验结果。
- 系统不会自动执行生成 SQL。
- 系统能拦截非只读 SQL。
- 用户确认后，系统可返回表格结果。
- 审计日志可查看历史问题、SQL、是否澄清、执行状态、耗时和错误信息。
- 生成与执行通过 `audit_log_id` 精确绑定。

## 13. 风险点

- `Oracle` 会提高数据库适配与测试成本。
- Hermes 输出质量依赖 SQLite Schema 完整性、字段备注质量、Obsidian 知识卡片质量和业务口径清晰度。
- 本地 Hermes CLI 和 Obsidian 路径属于本机环境依赖，需要清晰的设置页和错误提示。
- 知识库过期检测目前基于同步时间，无法完全识别所有语义层面的口径变化。
- 首版只做本地单账号登录，不提供集中化账号治理、细粒度授权和合规审计能力。
- Token 保存在本地浏览器存储中，需要避免在不可信环境中使用。
- 表命名和字段备注质量会影响 Hermes 对用途与关联关系的总结准确性；推测性关联需要在知识卡片中明确表达不确定性。

## 14. 技术栈

当前首版采用：`Vue 3 + TypeScript + Naive UI + FastAPI + SQLModel + SQLite + RSA + Token + Hermes CLI + Obsidian`

- `Vue 3 + TypeScript`：构建本地 Web 管理与问答界面。
- `Naive UI`：承载表单、数据表格、状态标签和企业工具型界面。
- `FastAPI`：承载数据源连接、Schema 同步、知识库同步、Hermes 编排、SQL 校验、执行与日志。
- `SQLite`：保存配置、Schema、备注、用户、登录日志、知识库同步任务和审计日志。
- `RSA + Token`：承载登录密码传输加密和业务请求鉴权。
- `Hermes CLI`：在受控 Schema Context 和相关业务说明内完成澄清、关系补全和 SQL 生成。
- `Obsidian Markdown`：按数据源组织人类可读的本地知识卡片，作为问答补充上下文。
