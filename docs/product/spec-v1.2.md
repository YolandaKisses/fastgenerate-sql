# 企业内部 SQL 问答 Web 应用 Spec

**版本**：Spec v1.2  
**状态**：当前工程基线版  
**更新时间**：2026-05-02

## 1. Spec 范围定义

本 spec 覆盖当前 Hermes / Obsidian 路线的首版 MVP。  
本 spec 定义产品行为、页面要求、状态变化、异常处理和验收口径。

## 2. 系统定位

系统是一款企业内部自用的 Web SQL 问答工具。系统通过本地 SQLite 保存数据源配置、Schema、备注、同步任务和审计日志；通过 Obsidian Markdown 知识库承载可读的表知识卡片；通过本地 Hermes CLI 完成知识库检索、澄清判断和 SQL 候选生成。

系统不是数据库管理工具，不承担写入、建表、改表、运维、发布、审批等职责。系统不是通用知识库问答工具，只围绕已配置数据源完成 SQL 问答。

## 3. 首版子系统拆分

1. 数据源管理
2. 本地登录与鉴权
3. Schema 同步与备注管理
4. Obsidian 知识库同步
5. 本地运行设置
6. Hermes SQL 问答
7. SQL 校验、执行与结果展示
8. 本地审计日志

## 4. 全局业务原则

- 所有配置均保存在本机。
- 用户必须先登录系统，业务页面和业务接口必须校验 Token。
- 工作台可以默认选中第一个符合问答准入条件的数据源；除此之外，系统不得根据问题内容自动切换数据源。
- 系统不得自动执行 Hermes 生成的 SQL。
- 所有执行行为都必须经过“只读校验”与“用户确认”两个关口。
- 首版使用本地 Hermes CLI，不接入外部 OpenAI 兼容模型。
- 首版不使用 embedding。
- SQLite 是运行时主存储，Obsidian 是 Hermes 问答链路的本地知识库载体。
- 首版结果展示以表格为主。
- 本地审计日志仅用于设备内回看与排查。

## 5. 本地存储规则

本地 SQLite 保存：

- 数据源配置
- 数据源连接状态与知识库同步状态
- 本地用户与登录日志
- Schema 表和字段
- 原始备注与本地补充备注
- 知识库同步任务
- 运行设置
- 审计日志

Obsidian 目录保存按数据源生成的 Markdown 知识库文件。Obsidian 文件可被 Hermes 读取，但不替代 SQLite 中的结构化数据。

## 6. 核心业务对象定义

### 6.1 数据源

本地保存的数据库连接配置，包含数据库类型、连接参数、显示名称、连接状态和知识库同步状态。

### 6.2 本地用户

本机登录账号，首版自动初始化一个管理员用户：用户名称 `系统管理员`，账号 `admin`，密码 `888888`。用户表保存自动生成的 `user_id`、账号、用户名称、角色、密码哈希、密码盐和状态，不保存明文密码。

用户表字段：

| 字段 | 说明 |
| --- | --- |
| `id` | SQLite 自增主键 |
| `user_id` | 对外用户 ID，使用 UUID 自动生成，唯一索引 |
| `name` | 用户名称，例如 `系统管理员` |
| `account` | 登录账号，例如 `admin`，唯一索引 |
| `password_hash` | 密码哈希 |
| `password_salt` | 密码盐 |
| `role` | 首版默认 `admin` |
| `is_active` | 是否启用 |
| `last_login_at` | 最近成功登录时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

密码哈希规则：

- 使用 `PBKDF2-HMAC-SHA256`。
- 迭代次数为 `210000`。
- Salt 使用随机值生成。
- 密码哈希与 Salt 分开保存。

### 6.3 登录日志

记录登录行为，包含账号、用户标识、是否成功、失败原因、客户端 IP、User-Agent 和发生时间，用于本机问题排查。

登录日志表字段：

| 字段 | 说明 |
| --- | --- |
| `id` | SQLite 自增主键 |
| `user_id` | 成功登录时记录用户 ID；失败且账号不存在时为空 |
| `account` | 用户输入账号 |
| `success` | 是否成功 |
| `failure_reason` | 失败原因，例如用户不存在、密码错误、用户已禁用、请求格式错误 |
| `ip_address` | 请求来源 IP |
| `user_agent` | 浏览器 UA |
| `created_at` | 登录时间 |

### 6.4 Schema

从数据库同步得到的元数据集合，包含表、字段、字段类型、数据库原始备注。

### 6.5 本地补充备注

用户在本机对表或字段补充的业务解释，与数据库原始备注分离存储，不回写数据库。

### 6.6 知识库任务

一次将数据源 Schema 与备注同步到 Obsidian 的任务，记录总表数、成功表数、失败表数、输出目录、任务状态和错误信息。

知识库任务字段：

| 字段 | 说明 |
| --- | --- |
| `id` | SQLite 自增主键 |
| `datasource_id` | 数据源 ID |
| `datasource_name` | 数据源名称快照 |
| `status` | `pending / running / completed / partial_success / failed` |
| `total_tables` | 总表数 |
| `completed_tables` | 已成功生成表数 |
| `failed_tables` | 生成失败表数 |
| `output_root` | Obsidian 根目录 |
| `output_dir` | 本次数据源输出目录 |
| `error_message` | 任务级错误信息 |
| `started_at` | 开始时间 |
| `finished_at` | 完成时间 |
| `created_at` | 创建时间 |

### 6.7 生成 SQL

Hermes 根据当前问题、对话历史和 Obsidian 知识库生成的候选 SQL，在用户确认前不可自动执行。

### 6.8 审计日志

本地留痕记录，包含提问、数据源、是否澄清、澄清内容、SQL、used notes、是否执行、执行状态、耗时、错误信息。

## 7. 本地登录与鉴权规则

### 7.1 登录规则

- 系统启动时必须保证本地管理员账号存在。
- 初始账号为 `admin`，初始密码为 `888888`。
- 登录页账号和密码必填。
- 登录成功后后端返回 Token 与用户信息。
- 登录成功默认跳转到数据源配置页；如果带有合法 `redirect` 参数，则优先回到原目标地址。
- 账号不存在、密码错误、用户禁用、请求格式错误都必须写入登录日志。

### 7.2 密码与 Token 规则

- 前端登录前先获取 RSA 公钥。
- 前端使用 `RSA-OAEP + SHA-256` 加密密码后提交。
- 后端解密后只用于校验密码哈希，不落库明文。
- RSA 密钥长度为 `2048`，密钥对由后端生成并持久化在本地数据目录。
- 私钥路径为 `{DATA_DIR}/auth/rsa_private.pem`，公钥路径为 `{DATA_DIR}/auth/rsa_public.pem`。
- Token 签名密钥优先读取环境配置；未配置时自动生成并保存到 `{DATA_DIR}/auth/token_secret`。
- Token 由后端签发，前端保存到本地浏览器存储。
- Token 默认有效期为 `480` 分钟。
- Token 使用 HMAC-SHA256 签名，格式为 `base64url(header).base64url(payload).base64url(signature)`。
- Token Header 包含 `alg = HS256`、`typ = FGSQL`。
- Token Payload 至少包含 `sub`、`account`、`role`、`iat`、`exp`。
- 普通业务请求必须携带 `Authorization: Bearer <token>`。
- 流式 SSE 请求必须使用 `fetch + ReadableStream` 携带同样的 Token。
- 无 Token、Token 无效或 Token 过期时，后端返回 `401`，前端清理本地登录态并返回登录页。
- 首版不做刷新 Token。

### 7.3 权限边界

- `/auth/public-key` 和 `/auth/login` 不需要 Token。
- `/auth/me` 需要 Token，用于读取当前登录用户。
- 业务接口必须校验 Token。
- 首版不做用户管理后台、组织级权限管理和 RBAC 权限矩阵。

### 7.4 认证接口

`GET /api/v1/auth/public-key`

返回 RSA 公钥与加密算法信息：

```json
{
  "algorithm": "RSA-OAEP",
  "hash": "SHA-256",
  "public_key": "-----BEGIN PUBLIC KEY-----\\n...\\n-----END PUBLIC KEY-----\\n"
}
```

`POST /api/v1/auth/login`

请求体：

```json
{
  "account": "admin",
  "password": "base64_rsa_encrypted_password"
}
```

成功响应：

```json
{
  "token": "signed-token",
  "token_type": "bearer",
  "expires_in": 28800,
  "user": {
    "user_id": "auto-generated-user-id",
    "name": "系统管理员",
    "account": "admin",
    "role": "admin"
  }
}
```

`GET /api/v1/auth/me`

请求头：

```http
Authorization: Bearer signed-token
```

成功响应：

```json
{
  "user_id": "auto-generated-user-id",
  "name": "系统管理员",
  "account": "admin",
  "role": "admin"
}
```

### 7.5 前端登录态规则

- Token 保存键名为 `fg_sql_token`。
- 用户信息保存键名为 `fg_sql_user`。
- 登录页只渲染登录内容，不显示业务侧边栏。
- 登录成功保存 Token 和用户信息。
- 退出登录时清除 Token 和用户信息，并跳转登录页。
- 收到业务接口 `401` 时清除本地登录态，并跳转登录页。

### 7.6 流式鉴权规则

- 原生 `EventSource` 不能设置自定义请求头，因此首版流式订阅统一使用 `fetch + ReadableStream`。
- 流式客户端按 SSE 协议解析 `event:` 和 `data:`，并按事件名分发。
- `AbortController.abort()` 用于停止当前流式读取。
- 工作台问答保留 `status / note_hit / note_used / hermes_trace / result / error` 事件。
- 知识库同步订阅保留任务启动与任务订阅分离，终态 `completed / partial_success / failed` 后停止读取。
- 该方案是单条 HTTP 长连接持续读取服务端推送，不是前端轮询。

## 8. 数据源管理规则

### 8.1 模块职责

- 新增数据源
- 编辑数据源
- 删除数据源
- 测试连接
- 展示连接状态
- 触发 Schema 同步
- 展示可用状态

### 8.2 状态定义

- `draft`
- `connection_failed`
- `connection_ok`
- `sync_failed`
- `stale`

### 8.3 核心规则

- 未填写必填字段不得保存。
- 保存成功不等于“可问答”。
- 当前实现中，只有同时满足 `status = connection_ok` 且 `sync_status = sync_success` 的数据源才可进入工作台问答。
- 编辑关键连接信息，包括 `db_type`，应重新标记为 `stale`。
- 删除数据源时，应同时删除与该数据源相关的本地 Schema 索引和补充备注。
- 历史审计日志应保留文本快照，不因数据源删除而丢失可读性。
- 普通用户名密码连接模式下，数据源数据库密码使用本地 AES-GCM 密钥加密后保存到 SQLite；服务启动时会迁移旧的明文数据源密码。SSH 通道模式不采集数据库密码时，该字段可为空。

## 9. Schema 同步与备注管理规则

### 9.1 核心规则

- Schema 同步不调用 Hermes。
- 通过数据库系统表或信息视图直接获取元数据。
- 必须区分“数据库原始备注”和“本地补充备注”。
- 本地补充备注不得回写数据库。
- 同步数据库原始备注时，不得覆盖本地补充备注。
- 重新同步 Schema 时，数据库中已删除的表和字段必须从 SQLite 中清理；对应已删除表的 Obsidian Markdown 文件也必须删除，避免旧知识继续被检索。
- 已完成知识库同步的数据源再次同步 Schema 后，`sync_status` 必须回到非成功态，提示用户重新同步知识库。
- 若同步结果为空，系统不得将该数据源标记为可问答。

### 9.2 `sync_status` 当前实现口径

当前代码中的 `sync_status` 由 Schema 同步和知识库同步共同使用，但以知识库同步状态作为工作台准入依据：

- `never_synced`：尚未完成知识库同步；Schema 同步成功后也会保持该状态，用于提示继续同步到知识库。
- `syncing`：正在同步 Schema 或知识库。
- `sync_success`：知识库同步成功，且连接状态为 `connection_ok` 时允许进入问答。
- `sync_failed`：Schema 或知识库同步失败。
- `sync_partial_success`：知识库部分同步成功，当前工作台不将其视为可问答状态。

### 9.3 同步失败规则

当前实现要求问答前完成知识库同步。Schema 同步失败或知识库未同步成功时，用户仍可继续在数据源和 Schema 页面维护配置与备注，但工作台不会将该数据源列为可问答选项。

## 10. 知识库同步规则

### 10.1 前置条件

- 数据源存在。
- 当前数据源已有同步过的 Schema 表。
- Obsidian 根目录可用。
- Hermes CLI 可用。
- 知识库同步由后端后台任务执行，启动接口必须快速返回任务信息。
- 任务执行不得依赖 SSE 连接存活；前端刷新后可以重新订阅进行中的任务。

### 10.2 输出结构

知识库按数据源分目录输出：

- `index.md`
- `tables/{table_name}.md`

输出目录规则：

- 知识库根目录来自本地运行设置 `obsidian_vault_root`；未配置时才回退到环境默认值。
- 数据源名和表名都要做文件系统安全清洗。
- 中文名称可以保留，但必须剔除 `/`、`:`、`*`、`?`、`<`、`>`、`|` 等非法路径字符。
- 即使文件名经过清洗，Markdown 正文仍保留原始数据源名和表名。
- 重复同步同一数据源时，覆盖该数据源目录下已有文件。

数据源总览页 `index.md` 必须包含：

- 数据源名称
- 数据源类型
- 生成时间
- 表数量
- 任务或来源说明
- 表目录与 Obsidian wiki 链接
- 一段说明性简介

表卡片必须包含：

- 基础信息
- 用途说明
- 核心字段解读
- 关联笔记
- 常见关联关系
- 注意事项
- 字段明细

字段明细必须来自 SQLite 中的真实 Schema，不得由 Hermes 自由生成。

单表详情页必须保留返回数据源总览页的链接，并以结构化表格展示字段名、类型、原始备注和补充备注。

Markdown 结构参考：

```md
# <数据源名称>

## 数据源信息
- 类型: mysql
- 表数量: 34
- 生成时间: 2026-04-30 10:30

## 表目录
- [[tables/users]]

## 说明
本知识库由 FastGenerate SQL 基于本地已同步 Schema、原始备注与补充备注自动生成。
```

```md
# users

[[../index|返回数据源总览]]

## 基础信息
- 原始备注: 用户主表
- 补充备注: 存储业务注册用户，不含游客

## 用途说明

## 核心字段解读

## 常见关联关系

## 注意事项

## 字段明细
| 字段名 | 类型 | 原始备注 | 补充备注 |
| --- | --- | --- | --- |
```

### 10.3 任务状态

- `pending`
- `running`
- `completed`
- `partial_success`
- `failed`

### 10.4 Hermes 生成约束

Hermes 在逐表生成时接收结构化上下文：

- 数据源名称
- 数据源类型
- 当前表名
- 当前表原始备注
- 当前表补充备注
- 当前表全部字段、类型、原始备注、补充备注
- 可选的少量兄弟表名和备注，用于辅助推测关联关系

Hermes 输出只允许用于以下 Markdown 段落：

- 用途说明
- 核心字段解读
- 常见关联关系
- 注意事项

Prompt 约束：

- 不得捏造无法从表结构中推断出的业务事实。
- 不确定时必须保守表达。
- 关联关系如果属于推测，需要明确写出“推测”或“可能”。
- 输出应简洁、可读，优先使用中文。

### 10.5 进度反馈

- 每处理完一张表后更新任务进度。
- 进度至少包含 `total_tables / completed_tables / failed_tables`。
- 可选记录当前正在处理的表名。
- 前端通过 SSE 订阅任务进度，展示类似 `正在生成 3 / 18 张表` 的文案。

### 10.6 接口规则

知识库同步接口挂在 schema 路由下：

- `POST /api/v1/schema/knowledge/sync/{datasource_id}`：创建同步任务并启动后台执行。
- `GET /api/v1/schema/knowledge/tasks/{task_id}`：查询任务状态。
- `GET /api/v1/schema/knowledge/tasks/{task_id}/events`：通过 SSE 订阅任务进度。
- `GET /api/v1/schema/knowledge/status/{datasource_id}`：用于刷新后查找当前数据源的知识库任务状态。

后续可扩展但首版不要求：

- `POST /api/v1/schema/knowledge/regenerate-table/{table_id}`
- `POST /api/v1/schema/knowledge/retry/{task_id}`

### 10.7 失败处理

- Obsidian 目录不可写时，任务失败。
- Obsidian 根目录不存在时，任务失败。
- 数据源下没有任何已同步表时，任务失败。
- Hermes 配置缺失时，任务失败。
- Hermes 不存在、不可执行、超时或返回非法 JSON 时，当前表失败。
- 单表失败不得中断后续表。
- 所有表失败时任务为 `failed`。
- 部分表失败时任务为 `partial_success`。
- 任务应记录 `completed_tables` 和 `failed_tables`。
- 文件写入失败时，任务记录可读错误信息。
- 失败时保留已经生成的部分文件，不主动回滚。
- 重试时采用确定性覆盖策略。

## 12. SQL 问答规则

### 12.1 问答前置条件

- 已选择数据源；若存在可问答数据源，工作台可默认选中第一个符合条件的数据源。
- 数据源连接状态为 `connection_ok`。
- 数据源知识库同步状态为 `sync_success`。
- 当前数据源已有可用知识库目录。
- Hermes CLI 可用。

### 12.2 输出类型

问答模块首版只允许两种主输出：

- `clarification`
- `sql_candidate`

### 12.3 澄清优先规则

当出现以下情况时，系统必须先澄清而不是生成 SQL：

- 多个表都可能满足问题目标
- 同一业务词可能映射多个字段
- 统计口径不明确
- 时间边界不明确
- 知识库检索结果无法稳定收敛
- 问题语义过于模糊

澄清问题应尽量给出具体选项。用户回答澄清后，系统应携带对话历史继续生成 SQL。

### 12.4 Hermes 约束机制

系统调用 Hermes 时必须通过：

1. 程序前置规则判定
2. Prompt 强约束
3. 返回结果结构校验

Hermes 返回结果必须是 JSON。返回结构不合法时，系统应拦截并提示错误。

## 13. SQL 执行与结果展示规则

### 13.1 核心规则

- 所有 SQL 在执行前都必须经过只读校验。
- 系统永不自动执行 SQL。
- SQL 只有在用户手动点击执行后才可进入执行阶段。
- 校验失败的 SQL 必须禁止执行。
- 结果展示以表格为主。
- 若结果为空，应明确展示“无结果”。

### 13.2 安全边界

首版必须禁止：

- `INSERT`
- `UPDATE`
- `DELETE`
- `DROP`
- `ALTER`
- `TRUNCATE`
- `CREATE`
- 多语句执行
- 有副作用的存储过程或等价调用

### 13.3 结果保护策略

- 最大返回行数：`500`
- 最大执行时长：`30 秒`

### 13.4 手工编辑规则

首版不开放 SQL 手工编辑执行。

## 14. 本地审计日志规则

### 14.1 日志记录原则

- 日志记录是默认能力。
- 日志用于回看、排查、复现问题。
- 首版日志仅本地查看，不导出、不远程同步。
- 生成与执行通过 `audit_log_id` 精确绑定。

### 14.2 建议记录字段

- 时间
- 数据源名称快照
- 用户问题文本
- 是否触发澄清
- 澄清问题内容
- 生成 SQL
- 使用过的知识笔记
- 是否执行
- 执行结果状态
- 耗时
- 错误信息摘要

## 15. 页面定义

### 15.1 登录页

目标：完成本地账号登录，并在登录成功后进入数据源配置页或原目标页面。

关键模块：

- 账号输入
- 密码输入
- 密码可见切换
- 登录按钮
- 登录错误提示
- 加载状态

### 15.2 工作台

目标：完成“选数据源 -> 提问 -> 澄清 -> 生成 SQL -> 校验 -> 内联确认 -> 执行 -> 查看结果”的单页闭环。

关键模块：

- 数据源选择区
- 问题输入区
- Hermes 工作过程区
- 澄清问题区
- SQL 候选展示区
- SQL 校验结果区
- 内联确认区
- 结果表格区

### 15.3 数据源管理页

目标：管理本地数据源配置，并明确数据源当前状态。

### 15.4 Schema / 备注管理页

目标：查看元数据、维护本地补充备注、触发 Schema 同步和知识库同步。

### 15.5 本地运行设置页

目标：配置 Hermes CLI 路径和 Obsidian 根目录，并提供可用性测试。

### 15.6 审计日志页

目标：在本地查看历史问答与执行记录，用于回看和排查。

## 16. 异常处理原则

- 所有失败必须给出用户可理解的错误反馈。
- 登录失败、Token 失效和权限校验失败必须给出明确状态，并引导回登录页。
- 连接失败、Schema 同步失败、知识库同步失败、Hermes 失败、SQL 校验失败、SQL 执行失败要分开定义。
- 失败后尽量保留用户输入和上下文。
- Hermes 生成成功但 SQL 校验失败时，仍然保留生成 SQL 供用户查看，但禁止执行。
- 日志写入失败不能导致主流程静默成功。
- 知识库过期时提示用户重新同步，但不强制阻断问答。

## 17. 全局验收标准

- 支持本地管理员账号 `admin / 888888` 登录。
- 支持登录密码 RSA 加密传输。
- 支持 Token 鉴权，业务页面和业务接口无 Token 不可访问。
- 支持普通请求和流式请求携带 Token，流式请求不得退回前端轮询。
- 登录成功默认进入数据源配置页。
- 支持 `MySQL / PostgreSQL / Oracle`。
- 支持本机管理数据源。
- 支持 Schema 同步。
- 支持原始备注与补充备注分离。
- 支持本地运行设置。
- 支持 Obsidian 知识库同步。
- 支持知识库同步部分成功状态。
- 支持 Hermes 问答。
- 支持当前数据源内问答。
- 歧义场景先澄清。
- 只支持生成只读 SQL 候选。
- 永不自动执行 SQL。
- 用户确认后才执行。
- 结果以表格展示。
- 本地日志可查看。
- 生成与执行日志可精确绑定。
