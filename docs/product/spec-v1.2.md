# 企业内部 SQL 问答 Web 应用 Spec

**版本**：Spec v1.2  
**状态**：当前工程基线版  
**更新时间**：2026-04-30

## 1. Spec 范围定义

本 spec 覆盖当前 Hermes / Obsidian 路线的首版 MVP。  
本 spec 定义产品行为、页面要求、状态变化、异常处理和验收口径。

## 2. 系统定位

系统是一款企业内部自用的 Web SQL 问答工具。系统通过本地 SQLite 保存数据源配置、Schema、备注、同步任务和审计日志；通过 Obsidian Markdown 知识库承载可读的表知识卡片；通过本地 Hermes CLI 完成知识库检索、澄清判断和 SQL 候选生成。

系统不是数据库管理工具，不承担写入、建表、改表、运维、发布、审批等职责。系统不是通用知识库问答工具，只围绕已配置数据源完成 SQL 问答。

## 3. 首版子系统拆分

1. 数据源管理
2. Schema 同步与备注管理
3. Obsidian 知识库同步
4. 本地运行设置
5. Hermes SQL 问答
6. SQL 校验、执行与结果展示
7. 本地审计日志

## 4. 全局业务原则

- 所有配置均保存在本机。
- 用户必须先手动选择数据源，系统不得自动切换数据源。
- 系统不得自动执行 Hermes 生成的 SQL。
- 所有执行行为都必须经过“只读校验”与“用户确认”两个关口。
- 首版使用本地 Hermes CLI，不接入外部 OpenAI 兼容模型。
- 首版不使用 embedding。
- SQLite 是运行时主存储，Obsidian 是 Hermes 问答链路的本地知识库载体。
- 首版结果展示以表格为主。
- 本地审计日志仅用于设备内回看与排查。

## 5. 本地存储规则

本地 SQLite 保存：

- 数据源配置
- 数据源状态与同步状态
- Schema 表和字段
- 原始备注与本地补充备注
- 知识库同步任务
- 运行设置
- 审计日志

Obsidian 目录保存按数据源生成的 Markdown 知识库文件。Obsidian 文件可被 Hermes 读取，但不替代 SQLite 中的结构化数据。

## 6. 核心业务对象定义

### 6.1 数据源

本地保存的数据库连接配置，包含数据库类型、连接参数、显示名称、可用状态和同步状态。

### 6.2 Schema

从数据库同步得到的元数据集合，包含表、字段、字段类型、数据库原始备注。

### 6.3 本地补充备注

用户在本机对表或字段补充的业务解释，与数据库原始备注分离存储，不回写数据库。

### 6.4 知识库任务

一次将数据源 Schema 与备注同步到 Obsidian 的任务，记录总表数、成功表数、失败表数、输出目录、任务状态和错误信息。

### 6.5 生成 SQL

Hermes 根据当前问题、对话历史和 Obsidian 知识库生成的候选 SQL，在用户确认前不可自动执行。

### 6.6 审计日志

本地留痕记录，包含提问、数据源、是否澄清、澄清内容、SQL、used notes、是否执行、执行状态、耗时、错误信息。

## 7. 数据源管理规则

### 7.1 模块职责

- 新增数据源
- 编辑数据源
- 删除数据源
- 测试连接
- 展示连接状态
- 触发 Schema 同步
- 展示可用状态

### 7.2 状态定义

- `draft`
- `connection_failed`
- `connection_ok`
- `sync_failed`
- `ready`
- `stale`

### 7.3 核心规则

- 未填写必填字段不得保存。
- 保存成功不等于“可问答”。
- 只有通过连接测试的数据源才可问答。
- 编辑关键连接信息，包括 `db_type`，应重新标记为 `stale`。
- 删除数据源时，应同时删除与该数据源相关的本地 Schema 索引和补充备注。
- 历史审计日志应保留文本快照，不因数据源删除而丢失可读性。

## 8. Schema 同步与备注管理规则

### 8.1 核心规则

- Schema 同步不调用 Hermes。
- 通过数据库系统表或信息视图直接获取元数据。
- 必须区分“数据库原始备注”和“本地补充备注”。
- 本地补充备注不得回写数据库。
- 同步数据库原始备注时，不得覆盖本地补充备注。
- 若同步结果为空，系统不得将该数据源标记为 `ready`。

### 8.2 同步状态定义

- `never_synced`
- `syncing`
- `sync_success`
- `sync_failed`
- `sync_partial_success`

### 8.3 同步失败规则

只要数据源通过连接测试，即可进入问答流程。Schema 同步失败会影响知识库质量，但不自动禁止继续操作。

## 9. 知识库同步规则

### 9.1 前置条件

- 数据源存在。
- 当前数据源已有同步过的 Schema 表。
- Obsidian 根目录可用。
- Hermes CLI 可用。

### 9.2 输出结构

知识库按数据源分目录输出：

- `index.md`
- `tables/{table_name}.md`

表卡片必须包含：

- 基础信息
- 用途说明
- 核心字段解读
- 关联笔记
- 常见关联关系
- 注意事项
- 字段明细

字段明细必须来自 SQLite 中的真实 Schema，不得由 Hermes 自由生成。

### 9.3 任务状态

- `pending`
- `running`
- `completed`
- `partial_success`
- `failed`

### 9.4 失败处理

- Obsidian 目录不可写时，任务失败。
- Hermes 不存在、不可执行、超时或返回非法 JSON 时，当前表失败。
- 单表失败不得中断后续表。
- 所有表失败时任务为 `failed`。
- 部分表失败时任务为 `partial_success`。
- 任务应记录 `completed_tables` 和 `failed_tables`。

## 10. 本地运行设置规则

### 10.1 设置项

- `hermes_cli_path`
- `obsidian_vault_root`

### 10.2 核心规则

- 设置保存在 SQLite。
- 未配置时回退到环境默认值。
- 支持 `~` 用户目录展开。
- 页面提供 Hermes 测试按钮。
- 页面提供 Obsidian 目录检查按钮。

## 11. SQL 问答规则

### 11.1 问答前置条件

- 已手动选择数据源。
- 数据源状态允许问答。
- 当前数据源已有可用知识库目录。
- Hermes CLI 可用。

### 11.2 输出类型

问答模块首版只允许两种主输出：

- `clarification`
- `sql_candidate`

### 11.3 澄清优先规则

当出现以下情况时，系统必须先澄清而不是生成 SQL：

- 多个表都可能满足问题目标
- 同一业务词可能映射多个字段
- 统计口径不明确
- 时间边界不明确
- 知识库检索结果无法稳定收敛
- 问题语义过于模糊

澄清问题应尽量给出具体选项。用户回答澄清后，系统应携带对话历史继续生成 SQL。

### 11.4 Hermes 约束机制

系统调用 Hermes 时必须通过：

1. 程序前置规则判定
2. Prompt 强约束
3. 返回结果结构校验

Hermes 返回结果必须是 JSON。返回结构不合法时，系统应拦截并提示错误。

## 12. SQL 执行与结果展示规则

### 12.1 核心规则

- 所有 SQL 在执行前都必须经过只读校验。
- 系统永不自动执行 SQL。
- SQL 只有在用户手动点击执行后才可进入执行阶段。
- 校验失败的 SQL 必须禁止执行。
- 结果展示以表格为主。
- 若结果为空，应明确展示“无结果”。

### 12.2 安全边界

首版必须禁止：

- `INSERT`
- `UPDATE`
- `DELETE`
- `DROP`
- `ALTER`
- `TRUNCATE`
- `CREATE`
- 多语句执行
- 有副作用的存储过程或等价调用

### 12.3 结果保护策略

- 最大返回行数：`500`
- 最大执行时长：`30 秒`

### 12.4 手工编辑规则

首版不开放 SQL 手工编辑执行。

## 13. 本地审计日志规则

### 13.1 日志记录原则

- 日志记录是默认能力。
- 日志用于回看、排查、复现问题。
- 首版日志仅本地查看，不导出、不远程同步。
- 生成与执行通过 `audit_log_id` 精确绑定。

### 13.2 建议记录字段

- 时间
- 数据源名称快照
- 用户问题文本
- 是否触发澄清
- 澄清问题内容
- 生成 SQL
- 使用过的知识笔记
- 是否执行
- 执行结果状态
- 耗时
- 错误信息摘要

## 14. 页面定义

### 14.1 工作台

目标：完成“选数据源 -> 提问 -> 澄清 -> 生成 SQL -> 校验 -> 内联确认 -> 执行 -> 查看结果”的单页闭环。

关键模块：

- 数据源选择区
- 问题输入区
- Hermes 工作过程区
- 澄清问题区
- SQL 候选展示区
- SQL 校验结果区
- 内联确认区
- 结果表格区

### 14.2 数据源管理页

目标：管理本地数据源配置，并明确数据源当前状态。

### 14.3 Schema / 备注管理页

目标：查看元数据、维护本地补充备注、触发 Schema 同步和知识库同步。

### 14.4 本地运行设置页

目标：配置 Hermes CLI 路径和 Obsidian 根目录，并提供可用性测试。

### 14.5 审计日志页

目标：在本地查看历史问答与执行记录，用于回看和排查。

## 15. 异常处理原则

- 所有失败必须给出用户可理解的错误反馈。
- 连接失败、Schema 同步失败、知识库同步失败、Hermes 失败、SQL 校验失败、SQL 执行失败要分开定义。
- 失败后尽量保留用户输入和上下文。
- Hermes 生成成功但 SQL 校验失败时，仍然保留生成 SQL 供用户查看，但禁止执行。
- 日志写入失败不能导致主流程静默成功。
- 知识库过期时提示用户重新同步，但不强制阻断问答。

## 16. 全局验收标准

- 支持 `MySQL / PostgreSQL / Oracle`。
- 支持本机管理数据源。
- 支持 Schema 同步。
- 支持原始备注与补充备注分离。
- 支持本地运行设置。
- 支持 Obsidian 知识库同步。
- 支持知识库同步部分成功状态。
- 支持 Hermes 问答。
- 支持当前数据源内问答。
- 歧义场景先澄清。
- 只支持生成只读 SQL 候选。
- 永不自动执行 SQL。
- 用户确认后才执行。
- 结果以表格展示。
- 本地日志可查看。
- 生成与执行日志可精确绑定。
