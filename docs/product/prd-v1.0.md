# 企业内部 SQL 问答桌面应用 PRD

**版本**：PRD v1.1  
**状态**：已确认发布版

## 1. 背景

企业内部已经沉淀了大量业务数据库，但日常取数仍然依赖会写 SQL 的人员，导致查询效率低、沟通成本高、响应慢。  
本项目希望提供一款企业内部可用的桌面应用，让用户在手动确认数据源的前提下，通过自然语言生成 SQL、查看结果和回看历史记录。  
产品只聚焦 SQL 问答，不做企业知识库问答，不做通用 RAG，不做复杂 BI 平台。

## 2. 目标

- 用户可在桌面应用中手动选择一个数据源，并输入自然语言问题。
- 系统能够基于该数据源的表结构、字段和备注信息生成 SQL，并进行只读安全校验。
- 系统展示生成 SQL，由用户手动确认后再执行，系统永不自动执行生成 SQL。
- 系统执行 SQL 后返回表格结果。
- 系统在本地保留问答、SQL、执行结果与错误信息，便于回看和排查。
- 首版仅支持第三方模型 API，采用 OpenAI 兼容协议。
- 首版不使用 embedding 检索，采用关键词和规则方式召回相关 Schema。

## 3. 非目标

- 不做登录。
- 不做 workspace、多租户、RBAC 权限体系。
- 不做自动推荐数据源。
- 不做跨数据源联合查询。
- 不做图表为主的分析体验。
- 不做数据库写入、DDL、DML 操作。
- 不做本地大模型和本地 embedding 模型部署。
- 不做企业知识库、文档问答、多模态问答。
- 不做自动执行 SQL。

## 4. 目标用户

- 内部分析、运营、产品、财务等有取数需求、但不熟悉 SQL 的员工。
- 少量维护人员负责配置数据源、同步表结构和维护备注。

## 5. 用户场景

- 用户打开桌面应用，手动选择“销售库”。
- 输入“上周华东区销售额是多少”。
- 系统在当前数据源中检索相关表结构和字段，生成 SQL，完成只读校验后展示给用户。
- 用户确认 SQL 无误后手动点击执行。
- 系统返回表格结果，用户可回看本次问题、SQL、执行状态和错误信息。

## 6. 核心流程

1. 维护人员配置数据源连接。
2. 系统连接数据库并同步表结构、字段、字段类型、表备注和字段备注。
3. 系统将同步到的 Schema 信息保存到本地索引库。
4. 用户打开应用并手动选择数据源。
5. 用户输入自然语言问题。
6. 系统在当前数据源范围内，基于表名、字段名、表备注、字段备注做关键词匹配和规则检索，召回相关 Schema。
7. 系统将问题与召回的 Schema 上下文发送给第三方 OpenAI 兼容模型 API。
8. 模型返回 SQL。
9. 系统校验 SQL 是否为只读语句，并执行安全限制。
10. 系统展示 SQL，等待用户手动确认。
11. 用户点击执行后，系统执行 SQL 并返回表格结果。
12. 系统将问题、数据源、SQL、执行状态、耗时和错误信息写入本地审计日志。

## 7. 范围

### 7.1 MVP 包含

- 跨平台桌面应用，支持 Windows 和 macOS
- 数据源配置管理
- 支持 `MySQL / PostgreSQL / Oracle`
- 数据源连接测试
- Schema 同步
- 区分“数据库原始备注”和“本地补充备注”
- 单数据源 SQL 问答
- OpenAI 兼容协议模型配置
- SQL 展示
- 只读 SQL 校验
- 用户手动确认后执行 SQL
- 表格结果展示
- 本地审计日志页面

### 7.2 MVP 不包含

- 登录和用户体系
- 组织级权限管理
- 自动数据源推荐
- 多数据源联动
- 图表分析为主的展示模式
- embedding 检索
- 本地模型部署
- 集中化审计平台
- 高级脱敏和行级权限
- 日志导出

## 8. 边界条件

- 所有配置采用单机本地保存。
- 用户必须手动确认数据源，系统不自动切换数据源。
- 系统永不自动执行模型生成的 SQL。
- 所有 SQL 默认限制为只读查询。
- 首版结果展示以表格为主。
- 本地审计日志仅在当前设备内查看，不做服务端集中管理，也不支持导出。
- Schema 检索首版不使用 embedding，仅使用关键词与规则召回。
- 第三方模型调用统一按 OpenAI 兼容协议接入。

## 9. Schema 同步说明

Schema 同步不需要调用 LLM。  
系统通过数据库系统表或信息视图直接读取元数据，包括：

- 表名
- 字段名
- 字段类型
- 数据库原始表备注
- 数据库原始字段备注

系统还支持在本地维护“补充备注”，与数据库原始备注分开存储、分开展示。  
该过程是数据库元数据读取，不属于 AI 推理。

## 10. Schema 检索说明

首版不使用 embedding。  
系统仅在用户已选择的数据源范围内，基于表名、字段名、数据库原始备注、本地补充备注进行关键词匹配和规则检索，召回与当前问题最相关的 Schema。  
这些 Schema 将作为 SQL 生成时的上下文输入。  
后续版本可将 embedding 检索作为增强项引入，以提升同义词、业务口语和命名不统一场景下的召回准确率。

## 11. 验收标准

- 应用可在 Windows 和 macOS 上正常运行。
- 可新增、编辑、删除并测试 `MySQL / PostgreSQL / Oracle` 数据源。
- 系统可完成表结构、字段、原始备注同步。
- 系统支持维护本地补充备注，并与原始备注区分展示。
- 用户可手动选择一个数据源并完成自然语言到 SQL 的查询闭环。
- 系统能展示生成 SQL。
- 系统不会自动执行生成 SQL。
- 系统能拦截非只读 SQL。
- 用户确认后，系统可返回表格结果。
- 系统能在本地审计日志页查看历史问题、SQL、执行状态、耗时与错误信息。
- 切换数据源后，检索范围不会串库。

## 12. 风险点

- `Oracle` 会提高数据库适配与测试成本。
- Text-to-SQL 的实际效果高度依赖表命名、字段命名、备注质量和业务口径清晰度。
- 首版不做登录，因此本地审计日志更偏操作留痕和问题排查，不是严格合规审计。
- 第三方 OpenAI 兼容接口会带来出网边界问题，后续可能需要补充脱敏与发送策略。
- 如果后续直接参考或复用 `SQLBot` 较多源码，需要额外评估许可证限制。

## 13. 技术栈建议

推荐首版采用：`Tauri 2 + Vue 3 + TypeScript + Naive UI + FastAPI + SQLite`

- `Tauri 2`：适合做轻量跨平台桌面工具
- `Vue 3 + TypeScript`：开发效率高，适合从 0 到 1 快速落地
- `Naive UI`：适合桌面工具、表单、数据表格与高信息密度管理界面，且更容易做出克制的企业级质感
- `FastAPI`：适合做数据源连接、Schema 同步、SQL 校验、模型调用
- `SQLite`：适合本地保存配置、Schema 索引和审计日志
