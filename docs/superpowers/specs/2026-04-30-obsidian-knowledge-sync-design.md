# Obsidian 知识库同步设计

## 目标

当数据源成功完成表和字段同步，并且用户已经补充完表级与字段级备注后，系统提供一个新的操作，将当前数据源生成一套可直接用于 Obsidian 的本地知识库。

该能力需要满足：

- 按数据源分类生成 Markdown wiki 文件
- 通过 Hermes 为每张表补充知识总结增强内容
- 将文件直接写入设置页配置的本地 Obsidian 知识库目录
- 因为生成过程可能较长，前端需要能够看到任务进度

## 当前上下文

当前产品已经具备如下本地数据链路：

1. 数据源连接信息保存在本地
2. Schema 同步通过 Python 读取数据库元数据
3. 表和字段元数据写入本地 SQLite
4. 表级与字段级补充备注在前端维护后写回 SQLite
5. 工作台在生成 SQL 时，会把已同步的 Schema 与补充备注一起作为 Prompt 上下文

本次需求相关的现有模块包括：

- `api/app/services/schema_service.py`
- `api/app/api/routes/schema.py`
- `api/app/services/workbench_service.py`
- `src/pages/SchemaManager/index.vue`
- `src/pages/SchemaManager/components/SchemaEditor.vue`

这意味着 SQLite 目前已经是以下信息的本地事实来源：

- 数据源身份信息
- 已同步的表和字段
- 原始备注
- 补充备注

因此，新功能应该复用 SQLite 中已经沉淀好的 Schema 与备注，而不是让前端重新组装一整份上下文再提交给后端。

## 用户流程

1. 用户在 Schema 管理页选择一个数据源
2. 用户点击 `同步到知识库`
3. 前端调用 Python 后端接口
4. 后端立即创建一个知识库同步任务，并返回任务 id
5. 后端从 SQLite 读取该数据源下所有已同步的表、字段和备注
6. 后端逐表调用 Hermes 生成增强说明
7. 后端将 Markdown 文件写入配置好的 Obsidian 目录
8. 前端通过 SSE 订阅任务状态并展示进度
9. 任务完成后，前端展示成功、失败以及输出目录等信息

## 推荐架构

### 1. 前端触发与进度展示

在 Schema 管理页现有的同步按钮附近新增一个操作按钮：

- `重新同步该数据源`
- `同步到知识库`

该按钮只针对当前选中的数据源生效。

前端职责：

- 发起任务
- 展示 `待处理 / 进行中 / 成功 / 失败` 状态
- 通过 SSE 订阅任务状态
- 展示简洁进度信息，例如 `12 / 34 张表已生成`

前端不负责：

- 组装完整 Schema 上下文
- 直接调用 Hermes
- 直接写本地文件

### 2. 后端任务入口

在 schema 路由下新增接口：

- `POST /api/v1/schema/knowledge/sync/{datasource_id}`
- `GET /api/v1/schema/knowledge/tasks/{task_id}`

后续可扩展但首版非必须：

- `POST /api/v1/schema/knowledge/regenerate-table/{table_id}`
- `POST /api/v1/schema/knowledge/retry/{task_id}`

创建任务接口需要负责：

- 校验数据源是否存在
- 校验该数据源下是否已经有同步好的表
- 在 SQLite 中创建任务记录
- 立即返回任务元数据
- 启动后台生成流程

### 3. 知识库同步服务

建议新增独立服务文件，例如：

- `api/app/services/knowledge_service.py`

职责包括：

- 从 SQLite 读取数据源、表、字段和备注
- 构建逐表的 Hermes Prompt
- 调用 Hermes
- 渲染 Markdown 内容
- 更新任务进度和最终状态

这部分逻辑不建议放进 `workbench_service.py`，因为文档生成和 SQL 问答属于不同职责。

### 4. 文件写入层

Markdown 渲染和文件系统写入建议集中在一个小的辅助层中，可以放在 `knowledge_service.py` 内部，也可以拆成单独 helper。

职责包括：

- 清洗数据源名和表名，确保可安全作为文件名
- 自动创建目录
- 以 UTF-8 写入 Markdown
- 保证重复执行时覆盖行为可预测

## 数据模型变更

建议新增一张 SQLite 表，例如 `KnowledgeSyncTask`。

建议字段：

- `id`
- `datasource_id`
- `datasource_name`
- `status`
- `total_tables`
- `completed_tables`
- `failed_tables`
- `output_root`
- `output_dir`
- `error_message`
- `started_at`
- `finished_at`
- `created_at`

建议状态枚举：

- `pending`
- `running`
- `completed`
- `failed`

这张表只用于记录任务状态，不需要把最终生成的 Markdown 正文存回 SQLite。

## 为什么仍然需要 SQLite

这里仍然需要 SQLite，但并不是为了再保存一份 Markdown。

SQLite 继续承担两类职责：

- 作为知识库生成的权威输入源
- 作为异步任务状态存储

最终生成的 Markdown 文件直接写入 Obsidian，不需要再冗余保存进 SQLite。

## Obsidian 输出目录结构

知识库根目录来自本地运行设置 `obsidian_vault_root`；未配置时才回退到环境默认值。

按数据源输出后的目录结构建议如下：

```text
<obsidian_vault_root>/
  <datasource_name>/
    index.md
    tables/
      <table_name_1>.md
      <table_name_2>.md
```

命名规则建议：

- 数据源名和表名都做文件系统安全清洗
- 即使文件名经过清洗，文件内容里仍保留原始表名
- 中文名称可以保留，但仍需剔除 `/`、`:`、`*`、`?`、`<`、`>`、`|` 等非法路径字符

## Markdown 结构设计

### 数据源总览页

`index.md` 建议包含：

- 数据源名称
- 数据源类型
- 生成时间
- 表数量
- 任务或来源说明
- 表目录与 Obsidian wiki 链接
- 一段说明性简介

建议结构：

```md
# <数据源名称>

## 数据源信息
- 类型: mysql
- 表数量: 34
- 生成时间: 2026-04-30 10:30

## 表目录
- [[tables/users]]
- [[tables/orders]]

## 说明
本知识库由 FastGenerate SQL 基于本地已同步 Schema、原始备注与补充备注自动生成。
```

### 单表详情页

每张表对应一个单独 Markdown，建议包含：

- 表名
- 原始备注
- 补充备注
- Hermes 生成的用途说明
- Hermes 生成的核心字段解读
- Hermes 生成的常见关联关系
- Hermes 生成的注意事项
- 结构化字段表格
- 返回数据源总览页的链接

建议结构：

```md
# users

[[../index|返回数据源总览]]

## 基础信息
- 原始备注: 用户主表
- 补充备注: 存储业务注册用户，不含游客

## 用途说明
...

## 核心字段解读
...

## 常见关联关系
...

## 注意事项
...

## 字段明细
| 字段名 | 类型 | 原始备注 | 补充备注 |
| --- | --- | --- | --- |
| id | bigint | 主键 | 用户唯一标识 |
```

## Hermes Prompt 设计

Hermes 在逐表生成时应接收结构化上下文，包括：

- 数据源名称
- 数据源类型
- 当前表名
- 当前表原始备注
- 当前表补充备注
- 当前表全部字段、类型、原始备注、补充备注
- 可选地附带少量兄弟表名和备注，辅助推测关联关系

Hermes 的输出目标应限制为以下几个部分：

- `用途说明`
- `核心字段解读`
- `常见关联关系`
- `注意事项`

Prompt 约束建议：

- 不要捏造无法从表结构中推断出的业务事实
- 如果不确定，只做保守表达
- 关联关系如果属于推测，需要明确写出“推测”或“可能”
- 尽量输出简洁、可读的中文内容

为了降低幻觉风险，Markdown 中的字段明细表必须始终来自 SQLite 的真实结构化数据，不能直接采用模型自由生成的字段内容。

## 后台执行策略

该功能建议通过后端后台任务执行，保证 `POST` 接口可以快速返回。

首版可接受方案：

- 使用 FastAPI 的后台任务机制

如果未来需要更高可靠性，可再演进为：

- 进程内任务队列
- 独立 worker

在当前本地单用户部署前提下，首版使用后台任务已经足够。

## 进度反馈设计

任务进度建议按“每处理完一张表”更新一次：

- `total_tables`
- `completed_tables`
- 可选记录当前正在处理的表名

前端轮询频率建议为 1 到 2 秒一次。

典型进度文案示例：

- `正在生成 3 / 18 张表`
- `已完成 18 / 18 张表，正在生成总览页`
- `同步到知识库失败：Hermes 返回格式异常`

## 异常处理

需要明确处理以下失败场景：

1. Obsidian 根目录不存在
2. 数据源下没有任何已同步表
3. Hermes 配置缺失
4. Hermes 请求超时或返回非法内容
5. 文件写入失败，例如路径非法或权限问题

推荐行为：

- 将任务标记为失败，并保存可读错误信息
- 如果失败发生时已经生成了部分文件，保留已生成结果，不主动回滚
- 重试时采用确定性覆盖策略

## 覆盖与幂等策略

首版建议策略：

- 同一数据源重复执行时，直接覆盖该数据源目录下已有文件
- 每次任务都写入同一个数据源目录
- 文件内容重新完整生成，而不是尝试做局部合并

这样做的好处：

- 心智模型简单
- 不会产生重复页面
- 备注更新后能稳定反映到知识库中

可选后续增强：

- 为文件增加 frontmatter 记录生成时间等元数据
- 按时间戳保留历史快照目录

## 测试策略

### 后端测试

建议覆盖：

- 有效数据源时可以成功创建任务
- 数据源无同步表时正确拒绝
- 总览页和单表页 Markdown 渲染结果
- 文件名清洗逻辑
- 任务进度更新逻辑
- Hermes 失败或文件写入失败时的异常处理

单测中应 mock 掉 Hermes 调用。

### 前端测试

建议覆盖：

- 页面出现 `同步到知识库` 按钮
- 点击后发起任务请求
- 展示进行中进度
- 展示成功与失败状态

## 范围边界

本次功能包含：

- 以数据源为粒度的知识库同步
- 数据源总览页和单表详情页
- Hermes 驱动的知识总结增强
- 任务进度展示
- 本地 Obsidian 文件输出

本次功能不包含：

- 在 UI 中按单表重新生成
- 增量 diff 生成
- 在应用内编辑 Markdown
- 同步图片、图表或附件
- 多知识库切换管理

## 推荐实施顺序

1. 新增任务模型及初始化逻辑
2. 新增知识库同步接口
3. 实现从 SQLite 读取 Schema 与备注的服务逻辑
4. 实现 Markdown 渲染和文件写入
5. 接入 Hermes 逐表增强生成
6. 打通后台执行和任务状态更新
7. 增加前端按钮与进度展示
8. 补充测试

## 本设计中已经确认的决策

- 采用后端任务执行，而不是前端编排
- SQLite 继续作为 Schema 和任务状态的事实来源
- 不将生成出的 Markdown 正文存回 SQLite
- 同时生成数据源总览页和单表详情页
- 输出到设置页配置的 `obsidian_vault_root`
- 按数据源进行目录分类
- 重复生成时覆盖同一数据源已有输出

## 风险

- 当表命名不清晰时，Hermes 可能会对关联关系做出不准确推断
- FastAPI 后台任务适合本地单用户场景，但不适合更复杂的多进程任务调度
- 如果数据源表很多，单次任务耗时可能会比较长

结合当前“本地部署、单用户使用”的范围，这些风险是可以接受的。
